# 1차 단위 프로젝트 가이드라인

## 1. 프로젝트 주제 및 담당자

권장 그룹 디렉터리:

```text
group_a/  주제 1 작업 공간
serar_locale/  주제 2 작업 공간
faq_crawling/  주제 3 작업 공간
```

각 그룹은 담당 주제별 소스 코드, 작업 노트, 스크린샷, 중간 산출물을 각자의 디렉터리에 저장할 수 있습니다. 공통 데이터베이스 스키마, 공통 유틸리티, 최종 문서, 공통 Streamlit 통합 코드는 `src/`, `docs/`, `data/` 같은 프로젝트 공통 디렉터리에 둘 수 있습니다.

최종 프로젝트에서는 `group_a`, `serar_locale`, `faq_crawling`에서 각각 구현한 세 개의 프로그램을 하나의 최종 프로그램으로 통합해야 합니다. 각 그룹은 독립적으로 개발하되, 최종 산출물은 공통 데이터베이스, 공통 설정, 하나의 Streamlit 실행 진입점에서 세 가지 기능을 모두 사용할 수 있어야 합니다.

### 주제 1: 해당 지역 충전소 목록 및 상세 정보

담당자: 이수연, 최지흠

필수 기능:

- 선택한 지역의 충전소 목록을 보여줍니다.
- 충전소 위치를 지도에 표시합니다.
- 지도 위 또는 지도 주변에 그래프 기반 요약 정보를 제공합니다.
- 충전소를 클릭하거나 선택하면 상세 정보를 보여줍니다.

예상 정보 예시:

- 충전소명
- 주소
- 위도 및 경도
- 충전기 타입
- 운영 기관
- 운영 시간
- 주차 가능 여부, 제공되는 경우
- 연락처, 제공되는 경우

### 주제 2: 실시간 사용 가능 충전기 조회

담당자: 김일환, 정현두

필수 기능:

- 실시간 충전기 사용 가능 여부를 조회합니다.
- 리스트 뷰 또는 그래프 뷰를 주요 선택 화면으로 제공합니다.
- 리스트 항목 또는 그래프 요소를 선택하면 현재 사용 가능한 충전기 목록을 출력합니다.

예상 정보 예시:

- 충전소명
- 충전기 ID
- 충전기 타입
- 충전기 상태
- 사용 가능 충전기 수
- 마지막 업데이트 시간
- 지역 또는 구/군

### 주제 3: 충전소 혼잡도 분석

담당자: 채수환, 최지흠

필수 기능:

- 사용자가 지역을 선택할 수 있어야 합니다.
- 지역별 충전기 사용량 또는 혼잡도를 분석합니다.
- 혼잡도 결과를 그래프로 보여줍니다.
- 데이터가 허용하는 범위에서 시간, 지역, 충전소, 충전기 타입별 혼잡도를 비교합니다.

예상 분석 예시:

- 사용 가능 충전기 비율
- 사용 중 충전기 비율
- 충전소별 혼잡도 점수
- 지역별 혼잡도 순위
- 타임스탬프 데이터가 있는 경우 시간대별 혼잡도 추이

## 2. 최종 필수 산출물

### 2.1 데이터베이스 설계 문서

필수 문서:

- 논리 ERD
- 물리 ERD

필수 도구:

- erdcloud.com

제출 요구사항:

- 논리 ERD와 물리 ERD를 각각 이미지로 캡처합니다.
- 캡처한 이미지를 프로젝트 저장소 안에 저장합니다.

권장 저장 위치:

```text
docs/erd/logical_erd.png
docs/erd/physical_erd.png
```

ERD에는 다음 내용이 명확히 포함되어야 합니다.

- 테이블
- 컬럼
- 기본키
- 외래키
- 관계
- 물리 ERD의 데이터 타입
- 필요한 경우 NULL 허용 여부

### 2.2 수집 데이터

수집한 데이터는 백업 파일로 내보내고 저장해야 합니다.

허용 파일 형식:

- CSV
- JSON
- XLSX
- SQL dump

목적:

- 인터넷 연결이 끊어졌을 때를 대비한 백업
- 수집한 원본 또는 가공 데이터의 증빙
- 반복 가능한 프로젝트 시연 및 검증

중요 규칙:

- 프로젝트는 저장된 데이터 파일에 직접 의존하지 않고 실행될 수 있어야 합니다.
- 저장된 데이터 파일은 백업 및 export 산출물입니다.
- 주요 애플리케이션 흐름은 데이터베이스를 통해 데이터를 수집, 파싱, 저장, 조회해야 합니다.

권장 저장 위치:

```text
data/raw/
data/processed/
data/export/
```

예시:

```text
data/raw/charging_stations_raw.json
data/processed/charging_stations_clean.csv
data/export/db_export.sql
```

### 2.3 데이터 파싱 프로그램

소스 코드가 포함되어야 합니다.

필수 역할:

- Open API, 웹 스크래핑 또는 둘 모두를 사용해 데이터를 수집합니다.
- 수집한 데이터를 데이터베이스에 저장 가능한 구조로 파싱합니다.
- 파싱한 데이터를 Python으로 MySQL에 저장합니다.
- Streamlit GUI에서 저장된 데이터베이스 데이터를 조회할 수 있게 합니다.

허용 수집 방식:

## 3. 데이터베이스 요구사항

데이터베이스는 가능한 한 공통 테이블을 통해 세 가지 주제를 모두 지원해야 합니다.

권장 핵심 테이블:

- `regions`
- `charging_stations`
- `chargers`
- `charger_status_logs`
- `station_congestion_stats`

권장 관계 개요:

```text
regions 1:N charging_stations
charging_stations 1:N chargers
chargers 1:N charger_status_logs
charging_stations 1:N station_congestion_stats
```

최소 데이터베이스 기대사항:

- 충전소 데이터와 충전기 데이터는 분리합니다.
- 실시간 상태 로그는 정적인 충전소 메타데이터와 분리하여 저장합니다.
- 지역 정보는 필터링을 지원할 수 있을 정도로 정규화합니다.
- 혼잡도 분석은 가능한 경우 상태 또는 사용 관련 데이터를 기반으로 계산합니다.

## 4. Streamlit GUI 요구사항

GUI는 Streamlit으로 브라우저에서 확인할 수 있어야 합니다.

최종 GUI는 세 그룹의 기능을 하나로 합친 통합 Streamlit 애플리케이션이어야 합니다.

필수 페이지 또는 섹션:

- 지역별 충전소 지도 및 상세 정보 뷰
- 실시간 사용 가능 충전기 조회
- 지역별 혼잡도 분석 그래프

권장 실행 명령:

```bash
streamlit run app.py
```

통합 방식 권장사항:

- `app.py`를 최종 실행 진입점으로 사용합니다.
- 사이드 메뉴 또는 탭으로 주제 1, 주제 2, 주제 3 기능을 이동할 수 있게 구성합니다.
- 각 그룹의 기능은 필요하면 `group_a`, `serar_locale`, `faq_crawling`에서 먼저 개발한 뒤, 최종적으로 `src/` 아래 공통 구조로 정리합니다.
- 데이터베이스 연결, 환경 변수, 공통 조회 함수는 중복 구현하지 않고 공유 모듈로 관리합니다.
- 최종 발표와 시연은 하나의 통합 프로그램 기준으로 준비합니다.

최소 UI 기대사항:

- 지역 선택 컨트롤
- 충전소 리스트 또는 지도
- 클릭 또는 선택 기반 상세 정보 패널
- 사용 가능 충전기 목록
- 혼잡도 그래프
- 데이터를 사용할 수 없을 때 명확한 오류 메시지
- 환경 변수를 통한 데이터베이스 연결 설정

## 5. 환경 및 설정

소스 코드에 비밀 정보를 하드코딩하지 않습니다.

로컬 설정은 `.env`를 사용합니다.

```text
DB_HOST=localhost
DB_PORT=3306
DB_USER=<DB_USERNAME>
DB_PASSWORD=<DB_PASSWORD>
DB_NAME=<DB_NAME>
API_KEY=<API_KEY>
```

실제 인증 정보가 포함된 `.env`는 커밋하지 않습니다.

대신 예시 파일을 제공합니다.

```text
.env.example
```

## 8. 완료 기준

프로젝트는 다음 조건을 만족하면 완료된 것으로 봅니다.

- 세 가지 주제 기능이 모두 구현되어 있습니다.
- 세 가지 주제 기능이 하나의 최종 Streamlit 프로그램 안에서 동작합니다.
- Streamlit GUI에서 데이터베이스 기반 결과를 조회할 수 있습니다.
- MySQL에 파싱된 충전소 및 충전기 상태 데이터가 들어 있습니다.
- 백업 데이터 파일은 존재하지만 일반 실행에는 필요하지 않습니다.
- 논리 ERD와 물리 ERD 스크린샷이 저장소에 저장되어 있습니다.
- 10~15분 분량의 발표 자료가 PPT와 PDF로 준비되어 있습니다.
- 데이터베이스가 이미 구축되어 있다면, 인터넷이 없어도 팀원이 애플리케이션을 시연할 수 있습니다.

## 9. 팀 합의 아키텍처

본 프로젝트는 세 주제를 각각 따로 실행하는 프로그램이 아니라, 최종적으로 하나의 Streamlit 애플리케이션으로 통합합니다.

최종 실행 흐름은 다음과 같습니다.

```text
공공데이터포털 OpenAPI 호출
→ XML 응답 수신
→ Python에서 필요한 컬럼만 파싱
→ 선택 컬럼 CSV 백업 저장
→ 선택 컬럼 MySQL 저장
→ Streamlit은 MySQL SELECT 결과만 사용
```

중요 규칙:

- 공공데이터포털 전기차 충전소 API는 JSON이 아니라 XML 응답을 기준으로 처리합니다.
- CSV 파일은 디버깅, 검증, 백업 목적입니다.
- Streamlit 애플리케이션은 CSV 파일을 fallback 데이터로 읽지 않습니다.
- Streamlit 애플리케이션은 API를 직접 호출하지 않습니다.
- Streamlit 애플리케이션은 MySQL에 저장된 데이터만 조회합니다.
- API에서 받은 모든 필드를 DB에 저장하지 않고, 프로젝트에서 사용하는 컬럼만 저장합니다.
- 단, API 필드 검증이 필요할 때는 별도 디버깅 export로 확인할 수 있습니다.

## 10. 컬럼 저장 정책

DB와 CSV 백업에는 프로젝트에 필요한 컬럼만 저장합니다.

권장 저장 기준:

- 화면에 직접 표시하는 컬럼
- 필터링, 정렬, 검색에 사용하는 컬럼
- 주제 2 사용 가능 충전기 조회에 필요한 컬럼
- 주제 3 혼잡도 계산에 필요한 컬럼
- 충전소, 충전기, 실시간 상태를 연결하는 식별키
- 데이터 수집 시점 확인을 위한 `fetched_at`

최소 핵심 식별키:

```text
statId   충전소 ID
chgerId  충전기 ID
```

`statId`, `chgerId` 조합은 충전기 상태와 충전기 기본정보를 연결하는 기준입니다. 이 값을 버리거나 별도 auto increment ID만 기준으로 삼으면 안 됩니다.

주제 2와 주제 3은 같은 실시간 상태 데이터를 공유합니다.

```text
주제 2: 지금 사용 가능한 충전기는 무엇인가?
주제 3: 지역/충전소/시간대별로 얼마나 혼잡한가?
```

따라서 `charger_status_logs`처럼 상태 이력을 저장하는 테이블이 필요합니다. 최신 상태만 덮어쓰면 주제 3의 시간대별 분석이 어려워집니다.

## 11. 권장 공통 소스 구조

최종 통합 구조는 다음 형태를 권장합니다.

```text
src/
  app.py                         # 최종 Streamlit 실행 진입점
  config.py                      # 환경변수, 프로젝트 경로, 공통 설정
  constants.py                   # 지역코드, 상태코드 등 상수
  api_fields.py                  # API에서 사용할 선택 컬럼 정의
  collect/
    ev_charger_client.py         # OpenAPI XML 호출 클라이언트
    export_selected_api_csv.py   # 선택 컬럼 CSV 백업 export
  parse/
    station_parser.py            # 충전소/충전기 기본정보 정리
    charger_status_parser.py     # 실시간 상태정보 정리
  queries/
    serar_locale_queries.py           # 주제 2 전용 SELECT 쿼리
    station_queries.py           # 충전소 공통 조회 쿼리
    congestion_queries.py        # 혼잡도 분석 쿼리
  service/
    serar_locale_availability.py      # 주제 2 비즈니스 로직
    station_service.py           # 주제 1 공통 로직
    congestion_service.py        # 주제 3 공통 로직
  ui/
    serar_locale_available_view.py    # 주제 2 Streamlit 화면
    map_view.py                  # 지도 화면 공통 컴포넌트
    chart_view.py                # 그래프 공통 컴포넌트
```

`group_a/`, `serar_locale/`, `faq_crawling/`는 작업 노트, 화면 캡처, 중간 산출물 보관 공간으로 사용합니다. 최종 실행에 필요한 공통 코드는 `src/` 아래로 이동합니다.

## 12. 주제 2와 주제 3의 역할 구분

주제 2와 주제 3은 같은 상태 데이터를 사용하지만 목적이 다릅니다.

주제 2:

- 현재 사용 가능한 충전기 목록을 보여줍니다.
- 충전기 단위 상세 정보가 중요합니다.
- 지역, 충전소명, 충전기 타입, 충전용량 기준 필터링을 제공합니다.
- 마지막 상태 갱신 시간을 보여줍니다.

주제 3:

- 지역별 또는 충전소별 혼잡도를 계산합니다.
- 충전기 개별 목록보다 집계 지표가 중요합니다.
- 사용 가능 비율, 사용 중 비율, 혼잡도 점수, 순위, 추이를 제공합니다.

공통 계산 예시:

```text
전체 충전기 수 = total_chargers
사용 가능 충전기 수 = available_chargers
사용 중 또는 사용 불가 충전기 수 = unavailable_chargers
사용 가능 비율 = available_chargers / total_chargers
혼잡도 점수 = 1 - 사용 가능 비율
```

이 공통 계산은 중복 구현하지 말고 `service/` 또는 `queries/`의 공통 모듈로 관리합니다.
